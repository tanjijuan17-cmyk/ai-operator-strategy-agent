# 设计思考
## 项目背景
这个项目基于 Dify Chatflow 搭建，核心目标是将用户的自然语言输入转化为结构化运营需求，并根据用户当前输入的信息完整度，动态决定后续流程是继续追问澄清，还是直接进入需求生成阶段。

在项目初期，我关注的重点并不仅仅是“如何生成结果”，而是希望进一步验证：在 AI Workflow 场景下，LLM 是否能够真正承担业务需求理解的能力，以及如何在模型生成能力与流程稳定性之间建立平衡。

相比传统规则式系统，这类 Workflow 的难点并不在于流程搭建本身，而在于如何处理 LLM 所带来的“不确定性”。因此，整个项目的设计重点，逐渐从“功能实现”转向了“职责边界”和“流程稳定性”的思考。

## 关于需求解析节点的设计思考
在最初的方案设计中，我曾尝试让同一个需求解析节点同时完成两项任务：一方面对用户输入进行语义解析，另一方面判断当前信息是否已经足够生成结构化运营需求。从 Workflow 结构上看，这种设计能够有效减少节点数量，使整体链路更加简洁。
这种方式在项目初期具有较高的实现效率，也能够快速完成基础流程验证。但随着测试场景逐渐增多，我开始意识到：当一个 LLM 节点同时承担“语义理解”与“流程决策”两类职责时，系统会产生明显的耦合问题。
